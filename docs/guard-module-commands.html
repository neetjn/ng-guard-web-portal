Module Commands Index

User Module

The User Module requires developers supply both a username and phrase with their request. The following is an example of a proper User Module request,

//driver?key=0&mod=user&user=username&phrase=phrase

and the following is a list of commands, 

    
    function \Module\User::data( $branch ) {
        /*
         * return profile data dump
         *
             
            expecting $branch as 'guard' || 'foro'
             
         */
    }
        
    function \Module\User::query( $field ) {
        /*
         * return guard user data
         *
             
            expecting $field as 'admin' || 'developer' || 'banned'
             
         */
    }
        
    function \Module\User::bit.read( $field ) {
        /*
         * return custom bit data
         *
              
            bit data structured using developer key as node
            
            $user_bit =
            {
                "key" :
                {
                    "field" : "value"
                }
            };
              
         */
    }
        
    function \Module\User::bit.update( $field, $update ) {
        /*
         * update custom bit data
         *
             
            bit data structured using developer key as node
            
            $user_bit =
            {
                "key" :
                {
                    "field" : "value"
                }
            };
             
         */
    }
        
    function \Module\User::subscribe( $tag ) {
        /*
         * subscribe to authorized subscription
         *
             
            information automatically queried from tag
             
         */
    }

These commands can be specified exactly as labeled above. As for passing parameters, they can be specified in alphabetical order ( `a`, `b`, `c`, `d`, `e`, `f` ). The maximal number of parameters read, are 6. An example of a complete request being,

//driver?key=0&mod=user&user=username&phrase=phrase&cmd=data&a=guard

`a` standing for `$branch`.

When using the User Module, developers can alternatively use 'sessions'. Sessions are used to deprecate the specification of a user's username and authentication phrase. This feature works by storing an encoded token in a web client or browser, whilst communicating with the main driver. A session can be created like so,

//driver?key=0&mod=user&sesh=create&a=username&b=phrase

Once a session is created, developers can use user commands without specifying a username or phrase,

//driver?key=0&mod=user&sesh=run&cmd=data&a=guard

Sessions can also be closed easily with the following command,

//driver?key=0&mod=user&sesh=end

If Sessions are not manually closed, they will be automatically deactivated after 24 hours.

App Module

Every App is bestowed an ID serial upon creation. These serials must be specified when creating a request. The following is an example of a proper App Module request,

//driver?key=0&mod=app&id=serial

and the following is a list of commands, 

    
    function \Module\App::data() {
        // return app data dump
    }
    
    function \Module\App::read( $field ) {
        // return app data
    }
    
    function \Module\App::update( $field, $update ) {
        /*
         * update data field
         *
         
            developer key must match with app key
            - app can be created @ http://guard.neetgroup.net/developers.doc
         */
    }
    
    function \Module\App::delete( $field ) {
        /*
         * completely remove data field
         *
         
            developer key must match with app key
            - app can be created @ http://guard.neetgroup.net/developers.doc
         
         */
    }

These commands can be specified exactly as labeled above. As for passing parameters, they can be specified in alphabetical order ( `a`, `b`, `c`, `d`, `e`, `f` ). The maximal number of parameters read, are 6. An example of a complete request being,

//driver?key=0&mod=app&id=serial&cmd=read&a=field

`a` standing for `$field`.

Util Module

The Util Module, is the only module that will function whilst the server is ever to go offline. Util Module requests also do not require and prereq parameters. The following is a list of commands, 

    
    function \Module\Util::online() {
        // return driver status
    }
    
    function \Module\Util::maintenance() {
        // return driver status
    }
    
    function \Module\Util::blacklisted( $gizmo_key ) {
        // x
    }

These commands can be specified exactly as labeled above. As for passing parameters, they can be specified in alphabetical order ( `a`, `b`, `c`, `d`, `e`, `f` ). The maximal number of parameters read, are 6. An example of a complete request being,

//driver?key=0&mod=util&cmd=blacklisted&a=key

`a` standing for `$gizmo_key`.
